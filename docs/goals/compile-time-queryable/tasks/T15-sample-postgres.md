# T15 — Canonical sample rewriter

Ship a sample rewriter: Postgres-style table → parameterized SQL + Dapper-style materializer, end-to-end.

Cite — `v1-deliverables`: *"Canonical sample rewriter — Postgres-style table → SQL + Dapper-style materializer (end-to-end)"*.

## Outcome

A canonical demonstration that exercises the full mechanism — `[CompileTimeQueryable]` (T01), `IQueryableChainRewriter` (T02), `ChainRewriteContext` (T03), feature flag (T04), discovery (T05), chain detection (T06), capture (T07), terminators (T08), invocation (T09), substitution (T10) — on one realistic data-access shape. The sample contains, at minimum:

- A runtime-side **table type** `Postgres9Table<T>` marked `[CompileTimeQueryable("Roslyn.Samples.CompileTimeQueryable.Postgres.Rewriter.PostgresChainRewriter, Roslyn.Samples.CompileTimeQueryable.Postgres.Rewriter")]` (rewriter type's namespace matches its host assembly, not the runtime stub's). The type is a thin wrapper over `IDbConnection` + table name, implementing enough `IQueryable<T>`/`IAsyncEnumerable<T>` surface to make both the **T11 decline-fallback path** (flag on, rewriter returns `null`) and the **T04 flag-off path** (chain detection skipped entirely) execute end-to-end through the runtime IQueryable plumbing. Those two fallback executions are part of what the sample proves, not unused appendages.
- An **analyzer-shaped rewriter assembly** `Roslyn.Samples.CompileTimeQueryable.Postgres.Rewriter` containing `PostgresChainRewriter : IQueryableChainRewriter`. It translates `Select` / `Where` / `Skip` / `Take` / `OrderBy` / `OrderByDescending` / `ThenBy*` followed by a terminator (`ToList`, `ToArray`, `Single`, `First`, `Count`, `Any`, `foreach`, `await ToArrayAsync`) into a parameterized SQL string and an `IDbConnection`-style execute call. Lambda bodies are walked with the `SemanticModel` from `ChainRewriteContext` to produce SQL fragments; captured locals from T07's capture list are bound to `@p0`, `@p1`, …, and emitted as a parameter object.
- A small **runtime materializer** in the runtime stub assembly (Dapper-style: open a `DbCommand`, bind parameters, read columns into the row type via a generated mapping). The sample writes its own minimal materializer rather than taking a Dapper dependency — Roslyn's build does not pull third-party NuGet for samples, and a ~100-line `IDataReader` row mapper is enough to demonstrate the materialization shape without obscuring the language feature.
- A **sample consumer project** with call sites exercising: (i) constant predicates, (ii) captured locals (`int minAge = 18; ... where u.Age >= minAge`), (iii) a projection (`select new { u.Id, u.Name }`), (iv) `foreach` as the terminator (no explicit materializer call), (v) the async `await ToArrayAsync()` path (gated on T12's resolution — see "Open questions"), and (vi) one chain the rewriter intentionally declines (e.g. `Select` with a method call it doesn't understand) to exercise the T11 fallback and prove the runtime IQueryable plumbing still works.
- A **README** at the sample root walking through what the compiler does at each call site (chain detection → rewriter input → substituted source → emitted IL excerpt), so a reviewer can read the sample as an end-to-end spec companion to T17.

The substituted call site at each rewritten chain is the rewriter's returned source text, parsed by `SyntaxFactory.ParseExpression`, type-validated against the chain's destination type (T10), and inlined — same substitution path as Quotable.

## Edge cases / risks

- **Sample must compile under the feature flag and degrade gracefully without it.** Both flag settings must produce semantically identical results for the same inputs — the runtime IQueryable path and the rewritten path return the same materialized data. The Roslyn-internal E2E test asserts this against an in-memory fake `IDbConnection` that records the parameterized SQL it would have run (and returns scripted rows); the *real-Postgres* assertion is in the downstream packaged consumer csproj that runs against a containerized Postgres, not in the Roslyn-internal test.
- **SQL dialect choice (Postgres) is illustrative.** The compiler feature must not learn anything Postgres-specific — no SQL keywords, parameter syntax, or escape rules in the binder. The sample's rewriter is the *only* place dialect knowledge lives; a sibling MySQL/SQLite rewriter would drop in as a peer sample with no compiler change. T17's spec doc should state this explicitly so reviewers don't conflate "this works for Postgres" with "this special-cases Postgres".
- **SQL injection / parameter binding demonstration.** The sample's `Where`/`Select` translation must route captured locals through `@p0`-style parameters, never string-concatenate them into the SQL text. The README calls this out as the demonstration of "captures arrive as a typed list, not as syntax to re-evaluate" — that's how T07's design avoids the `Expression<T>` re-evaluation tax and the injection footgun simultaneously. The E2E test asserts the substituted source contains `@p0`/`@p1` and *not* the captured literal value.
- **Captured-variable path coverage.** The sample must include at least one chain with no captures (purely literal predicate), one with a captured local, one with a captured field, and one with a captured parameter — these are the canonical shapes the Quotable walker enumerates (`BoundLocal`/`BoundParameter` plus member-access on `this`). T07's final enumeration may add or rename shapes; the rewriter source code must demonstrate handling whichever set T07 freezes, and this list must be re-checked against T07 before T15 lands.
- **Foreach-terminator path.** `foreach (var u in db.Users.Where(...).Select(...))` does not produce a `ToArray`/`ToList` call; the terminator is the `foreach` itself per T08. The rewriter's substituted source must produce something the binder can iterate (an `IEnumerable<T>`), not a materialized `T[]`. The sample exercises this path because it's the easiest place to get the substitution shape wrong.
- **Dapper dependency policy.** Take none. Write a minimal materializer in the runtime stub assembly. Reasons: (i) Roslyn samples don't pull third-party NuGet; (ii) Dapper's `Query<T>(connection, sql, param)` shape obscures the language feature; (iii) the materializer is ~100 lines for the column shapes the sample needs and is itself a useful artifact for spec readers. Call out "real consumers would substitute Dapper, EF Core compiled queries, or a hand-rolled mapper" in the README — the sample's materializer is illustrative, not prescriptive.
- **Decline path must be exercised in the sample.** One of the consumer's call sites must trigger T11 fallback (e.g. `Where(u => SomeMethod(u))` where the rewriter doesn't recognize `SomeMethod`). Without this, reviewers can't visually confirm graceful degradation is wired up. The E2E test asserts: rewriter returned `null`, no diagnostic emitted, runtime path executed.
- **Rewriter AQN drift.** The AQN string `[CompileTimeQueryable("...")]` is a fragile join between two assemblies. The sample README must call out: changing the rewriter's namespace or type name is a breaking change for downstream consumers. The E2E test pins the AQN in a constant and uses `nameof(PostgresChainRewriter)` only for the type-name half, to surface accidental renames.
- **Async terminator pulls in T12.** `await db.Users.Where(...).ToArrayAsync()` requires settling T12's `IAsyncEnumerable<T>` scope — without that, the sample's async example is illegal and must be cut. The fallback is: ship the sync examples in v1, add the async example as a v1.1 amendment after T12 resolves. The E2E test conditionally enables the async example based on a T12 resolution constant.
- **Branched chains pull in T13.** If the sample includes `var q = db.Users.Where(...); var arr = q.ToArray(); var n = q.Count();` (one local consumed by two terminators), T13's model choice (per-terminator vs whole-DAG vs linear-only) decides whether the sample compiles. Cut branched examples until T13 lands; v1 sample is single-terminator-per-chain only.
- **Sample bit-rot.** The substituted-source snapshot in the E2E test will drift if the rewriter's translation strategy changes (formatting, operator precedence parens, parameter naming). Parse both the expected and actual substituted source and compare via `SyntaxNode.NormalizeWhitespace().ToFullString()` so trivial formatting churn doesn't break the assertion. (Quotable's analogous tests verify IL via `CompileAndVerify(...).VerifyIL(...)` rather than substituted source; T15 needs a source-snapshot variant because the substituted source is itself the artifact being demonstrated.)

Otherwise, drop the async example from v1.
- **Branched-chain inclusion** depends on T13. The sample as currently scoped is linear-only (one terminator per chain); if T13 picks whole-DAG or per-terminator, the sample gets one extra example demonstrating it.
