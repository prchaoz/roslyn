# T10 — Source-text substitution

When the rewriter's returned source is syntactically valid at the chain site, substitute it inline. Otherwise wrap it in a synthesized private static method, passing a typed capture record holding the chain's captured variables.

Cite — `rewriter-output`: *"inline when valid, wrap in synthesized static method otherwise"*.

## Outcome

"Syntactically valid at the chain site" means the rewriter's returned text parses (via `SyntaxFactory.ParseExpression`) as a single `ExpressionSyntax` that can replace the terminator-rooted call expression — the same single-expression shape that `[Quotable]`'s inline path accepts. Anything that requires statements (declarations, `try`/`finally`, branching, `yield`, multi-statement bodies that the rewriter author cannot collapse into an IIFE) falls through to the wrapped path. The decision is mechanical: try inline first, and only synthesize the wrapper when `ParseExpression` returns diagnostics, when the parsed expression fails `Conversions.ClassifyImplicitConversionFromExpression` against the chain-site type, or when it references a name that does not resolve in the call-site binder.

The wrapped path synthesizes two members at the **containing type of the enclosing top-level method** (`SyntheticBoundNodeFactory.CurrentType`): a private `[CompilerGenerated]` static method whose body is the rewriter's text, and a private `[CompilerGenerated]` capture record (struct, no equality members — the wrapper does not consume them, and a struct avoids an extra heap allocation per chain) with one strongly-typed field per `CapturedVariable`. Naming mirrors the Roslyn convention used by `SynthesizedClosureEnvironment` / `SynthesizedClosureMethod` (via `GeneratedNames.MakeLambdaMethodName` / `MakeLambdaDisplayClassName`): a `<TopLevelMethodName>g__CompileTimeQueryable|<chainSiteOrdinal>_<n>` suffix for the method and `<>c__Captures<chainSiteOrdinal>` for the record. The `chainSiteOrdinal` comes from a per-method counter held on `TypeCompilationState`, mirroring how closure ids are issued by `ClosureConversion.Analysis`. Name-collision avoidance reuses the `<…>` angle-bracket prefix that user code cannot produce; collisions between rewriters on co-located chains are resolved by the ordinal.

## Files to touch

- `src/Compilers/CSharp/Portable/Binder/Binder_Conversions.cs` — the substitution site. T09 invokes the rewriter inside the chain-binding helper; T10 splits the result into the inline path (parse + `BindExpression` + identity-wrap, mirroring `[Quotable]`'s `BindWithIdentifierMapSuspended`) and the wrapped path. The wrapped path emits a `BoundCall` to the synthesized wrapper passing a `BoundObjectCreationExpression` for the capture record built from the `CapturedVariable[]` produced by T07.
- `src/Compilers/CSharp/Portable/Lowering/` — a new lowerer (proposed `CompileTimeQueryableSubstitutionRewriter.cs`, paired with `SynthesizedCompileTimeQueryableMethod` and `SynthesizedCompileTimeQueryableCaptureRecord` modelled on `SynthesizedClosureMethod` / `SynthesizedClosureEnvironment`) registers the synthesized definitions via `CompilationState.ModuleBuilderOpt.AddSynthesizedDefinition(...)`. `SyntheticBoundNodeFactory` provides the helpers for `Field`, `Parameter`, `Call`, and `New`; the binder hands the lowerer the parsed expression syntax, the destination type, the capture list, and the chain-site `SyntaxNode` for diagnostics attribution. Late-bound parsing of the wrapper body is delayed until method-body lowering — the body is stored on the synthesized method as the raw rewriter text + capture-record symbol, then parsed/bound against a binder rooted in the synthesized method's parameter scope, with `InMethodBinder.IdentifierMap` suspended exactly as on the inline path.
- Coordinate with **T09** (invocation): T09 produces the `(string source, ImmutableArray<CapturedVariable> captures)` pair plus the chain-site bound tree; T10 consumes that contract. Coordinate with **T11** (decline): if the rewriter returns `null`, T11 short-circuits before T10 runs — silent fallback to runtime, no synthesized members emitted.

## Edge cases / risks

- **Unspeakable capture types** — anonymous types and tuples with element names cannot be named in a synthesized field declaration. The capture record's fields must use the *symbol-level* `TypeSymbol`, not a source-text rendering, so the synthesized type can reference the anonymous-type symbol directly (the existing `SynthesizedClosureEnvironment` already does this through `SynthesizedFieldSymbolBase`). Tuple element names survive on the field's `TypeWithAnnotations`.
- **`ref` / `in` / `out` parameters and ref-struct captures** — a record (struct or class) cannot hold byref or ref-struct fields under today's rules. For ref-struct fields the synthesized record must itself be a `ref struct`, which then forbids it from crossing iterator/async boundaries. For `ref` / `in` / `out` parameters the wrapped path must reject (decline to wrap) and either stay inline-only or fall through to T11; this mirrors the constraint `SynthesizedClosureEnvironment` enforces today.
- **Iterators and async** — if the rewriter's text contains `yield` or `await`, the synthesized wrapper method must itself be declared `async` or as an iterator. The lowerer inspects the parsed body for `YieldStatementSyntax` / `AwaitExpressionSyntax` (cheap syntactic scan, no semantic binding) and sets the synthesized method's `IsAsync` / `IsIterator` flags before the state-machine rewriters run. Inline substitution into a non-async/non-iterator chain context cannot accept such bodies — they must wrap.
- **Generic context propagation** — the chain site's enclosing method (and its containing type) may be generic. The synthesized method must inherit the enclosing type's type parameters (so it can reference them in its capture-record fields and body) and add its own for any rewriter-introduced generics — same `TypeMap.ConcatMethodTypeParameters` / `AssignTypeMapAndTypeParameters` pattern as `SynthesizedClosureMethod`. The capture record likewise needs the enclosing type's type parameters when any captured variable's type mentions them.
- **PDB sequence points** — the synthesized wrapper must be attributed to the chain-site `SyntaxNode` (`SyntheticBoundNodeFactory.Syntax`) so step-into from the call site lands in the user's source, not in compiler-generated locations. Default to `[DebuggerStepThrough]` so step-over preserves chain semantics without trapping the developer in synthesized code; surface as an open question whether rewriter authors need finer control. The capture-record constructor takes no separate sequence point. Inline substitution preserves the chain-site span verbatim by virtue of identity-wrapping.

## Open questions

- **T13 (branched chains)** affects whether one substitution serves multiple terminators or whether each terminator gets its own synthesized wrapper. Per-terminator implies N wrappers and N capture records per branched DAG; whole-DAG implies one wrapper with a more complex returned shape. T10 is implementable today under "linear-only v1" (T13 option c) — branching falls through to T11 — and the wrapped-vs-inline split is unaffected. Surface only; do not resolve here.
