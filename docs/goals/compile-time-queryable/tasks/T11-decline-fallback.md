# T11 — Silent fallback on rewriter decline

When the rewriter returns `null`, silently fall back to runtime evaluation through the type's normal `IQueryable`/`IEnumerable` plumbing (if any).

Cite — `decline-behavior`: *"(b) Silent fallback — chain runs at runtime through the type's normal IQueryable/IEnumerable plumbing if any"*.

## Outcome

A `null` return from `IQueryableChainRewriter.Rewrite(ChainRewriteContext)` is **not** an error and **not** a diagnostic-bearing event. The compiler discards every bound node produced for the rewriter path (the captured chain `BoundCall` sequence, the synthesized `ChainRewriteContext`, any tentative substitution buffer) and re-binds the original invocation chain as if `[CompileTimeQueryable]` were absent — i.e. through the normal overload-resolution, conversion-classification, and argument-binding paths in `Binder_Invocation.BindInvocationExpression`. The terminator call (`ToArrayAsync`, `foreach`, `Single`, …) then runs at execution time against whatever the marked type's regular instance / extension methods resolve to, which in the canonical Postgres sample is the type's own `IQueryable<T>` / `IAsyncEnumerable<T>` interface plumbing.

The fallback is **silent by default** even when the rewriter declined for principled reasons (unsupported operator, opaque captured value, partial-type information). The rewriter MAY have called `context.Reporter.Report(...)` from inside `Rewrite` before returning `null` — those diagnostics survive on the `DiagnosticBag` exactly as the rewriter emitted them — but the *absence* of any rewriter diagnostic is the expected case and the compiler emits nothing of its own. This is the explicit contrast against the sibling [Quotable] feature, where a null return with zero reported diagnostics raises `ERR_QuoterDeclined` to keep call sites from invisibly miscompiling (`custom-expression-trees/plan.md` Step 8); here, "invisibly fall back to runtime" *is* the documented success mode.

## Files to touch

- `src/Compilers/CSharp/Portable/Binder/Binder_Invocation.cs` — the fallback path. After T09's rewriter invocation produces `null`, return from the chain-rewrite branch back into the normal `BindInvocationExpression` / `BindMethodGroupInvocation` flow at the *chain root's* terminator-call site (line 183 entry; line 698 `BindMethodGroupInvocation`). Mechanically this means rolling back the temporary `BindingDiagnosticBag` used for the rewriter attempt only if it is empty — any rewriter-reported diagnostics still need to be merged into the real diagnostic bag.
- `src/Compilers/CSharp/Portable/Binder/Binder_Conversions.cs` — no-op when null. The rewriter only fires for chains whose receiver static type is `[CompileTimeQueryable]`; chains aren't routed through a synthetic conversion the way Quotable lambdas are (no `ConversionKind.AnonymousFunction` analogue), so there is **no positive substitution wrap to do** on decline. The change is defensive: any new helper introduced by T10 (e.g. `CreateCompileTimeQueryableSubstitution`) must early-return its argument unchanged when the rewriter returned `null`, and must not allocate a `BoundConversion` for the chain.

## Dependencies

- **Built on T09** — the rewriter-invocation harness. T09 owns the call to `IQueryableChainRewriter.Rewrite`; T11 owns what the binder does with the returned `null`.
- **Coordinates with T08 (terminators)** — silent fallback assumes the original chain *is bindable* by the normal binder. If the marked type does not implement `IQueryable<T>` / `IAsyncEnumerable<T>` and exposes no instance / extension methods named `Where`/`Select`/`ToArrayAsync`/etc., the fallback re-binding will surface plain `ERR_NoSuchMember` / `ERR_BadArgType` diagnostics at the terminator site. That is the correct end-user signal ("you wrote `q.Where(x => x.Id > 5).ToArrayAsync()` against a marked type that has no `Where`"), but it is *not* a "rewriter declined" diagnostic — it is the ordinary binder error the user would have seen with the feature flag off.
- **Coordinates with T10 (substitution)** — T10 wraps the *non-null* substitution path. T11 is the disjoint branch. T10 and T11 must share the same "tentative rewriter scratch state" object so that on `null` the scratch is dropped cleanly.
- **Independent of T04 (feature flag)** — the flag gates whether the rewriter path is taken at all; the flag-off behavior is **identical** to T11's fallback. Don't conflate them: flag-off skips T05/T06/T07/T09 entirely; T11 fires only on `null` *after* the rewriter ran.

## Edge cases / risks

- **Marked type has no runtime provider.** A `[CompileTimeQueryable]` type that supplies *only* a rewriter and exposes no `IQueryable<T>` / `IEnumerable<T>` / `IAsyncEnumerable<T>` interface and no instance methods will fail to bind under fallback. The interview answer ends "*through the type's normal IQueryable/IEnumerable plumbing **if any**.*" — the "if any" is load-bearing. When there is no plumbing, the user gets the ordinary unbound-method diagnostics from `Binder_Invocation`. We **do not** translate that into a synthesized "rewriter declined and no fallback exists" error: doing so would require either (a) probing for `IQueryable<T>` implementation before deciding whether silence is appropriate, or (b) maintaining two diagnostic bags (the rewriter's reported ones + the fallback binder's). Both add coupling for an authoring failure mode the marked-type author can prevent at design time. Document in the spec (T17) that supplying runtime plumbing is the author's responsibility.
- **Intermittent decline based on rewriter-internal state.** A rewriter that returns non-null for input `X` in one binding pass and `null` for the same `X` in a later pass (e.g. lazily-loaded schema, externally cached parser) breaks every incrementality assumption Roslyn makes. The chain's bound shape changes between calls; semantic-model queries return inconsistent results; IDE caching corrupts. Specify in T17 that `Rewrite` must be a pure function of `ChainRewriteContext`. The compiler does not enforce this — but a rewriter-invocation cache (whose introduction belongs to T09's enrichment) keyed on context identity, treating the first answer as canonical for the compilation, makes non-determinism manifest as "first call wins" rather than "intermittent corruption." Flag the cache requirement on T09 if it is not already there.
- **No IDE / error-list signal when fallback occurs.** Per the explicit interview choice, the user receives **no** indicator that their chain runs at runtime instead of being rewritten. This is the intentional design: predictable behavior, no false alarms, no IDE squiggles to ignore. The cost is **discoverability** — a chain that silently regresses from compile-time rewrite to runtime evaluation (e.g. because the rewriter author added an unsupported operator, or because a refactor changed a captured value's type) shows up only in profiling / observed query plans, not in the build log. T18 benchmarks should include a "regression detector" recipe (compare emitted IL against a golden) so adopters can wire their own CI tripwire.
- **Interaction with the rejected `OnDecline` policy.** The interview surfaced option (c) "configurable per-call-site decline behavior" (e.g. `[CompileTimeQueryable(OnDecline = OnDecline.Warn)]`). It was **not chosen**. Do not add an opt-in warning hook in this task; the design is single-mode silent fallback. If future demand surfaces, the natural extension point is a new attribute property on `[CompileTimeQueryable]` plus a third branch in this same code path — but that is out of scope for v1.
- **Diagnostics already on the bag at decline time.** If the rewriter called `context.Reporter.Report(...)` *and then* returned `null`, the reported diagnostics must survive. The fallback path must not blanket-discard the rewriter's diagnostic bag — only the *binder-internal* tentative diagnostics from the rewriter's bound-tree scratch space are dropped. Practically: route rewriter-reported diagnostics through a `BindingDiagnosticReporter`-shaped adapter (mirroring `custom-expression-trees/plan.md` Step 8) that writes straight to the *real* outer `BindingDiagnosticBag.DiagnosticBag`, bypassing the scratch bag entirely.
- **Caching implication.** Whichever T09 deliverable introduces rewriter-result caching must cache `null` returns with the same lifetime as successful returns. Re-invoking the rewriter on the same chain in the same compilation must not happen, both for performance and for the determinism point above. Suggested cache key: `(rewriter type identity, chain root syntax span, captured-variables structural hash)`.

## Open questions

None expected; the interview answer is decisive. Two adjacent intersections are flagged for tracking, not resolution here:

- **T12 (IAsyncEnumerable scope)** — if T12 lands "async chains rooted at the marked type are in scope," T11's fallback must re-bind the `await foreach` / `await ToArrayAsync()` terminator against the marked type's `IAsyncEnumerable<T>` interface plumbing. No additional T11 work, but the test matrix grows.
- **T13 (branched chains)** — if T13 picks "linear-only v1: any branching → fallback per T11," then T11 becomes the *common* path, not the rare path. The diagnostics-on-fallback policy stays unchanged (silent), but T18 benchmarks must measure fallback throughput as a primary axis, not a tail.
