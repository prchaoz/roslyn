# T03 — Add `ChainRewriteContext`

Add `ChainRewriteContext` carrying chain syntax, scoped `SemanticModel`, ordered bound-call sequence, in-chain `LambdaExpressionSyntax`s, captured-variable list, `IDiagnosticReporter`, and a fresh-name generator helper.

Cite — `rewriter-input` *(unresolved — see T13)*: user noted *"I think about the tree of possible chains with branches"*.

## Outcome

`ChainRewriteContext` is the single input parameter handed to `IQueryableChainRewriter.Rewrite` (T02). It is the queryable analog of `QuotationContext` in the `[Quotable]` design and inherits the same construction rule: the compiler precomputes everything the rewriter is allowed to see, and the rewriter does no further re-binding or dataflow analysis of its own.

Public surface (namespace `Microsoft.CodeAnalysis.CSharp.CompileTimeQueryable`, all marked `[Experimental("RSEXPERIMENTAL00N")]`):

- `ChainSyntax` — the `ExpressionSyntax` rooted at the chain's terminator call site (the substitution target). Used as the diagnostic-location anchor and as the source node for `SemanticModel` queries against the chain.
- `SemanticModel` — scoped to the enclosing member. The chain itself is still mid-binding when the rewriter runs, so the public contract forbids calling `AnalyzeDataFlow` against any node inside the chain (same re-entrancy stack overflow documented in `custom-expression-trees/plan.md` step 5).
- `ChainCalls` — `ImmutableArray<ChainCall>`, ordered source→terminator. Each `ChainCall` exposes the `InvocationExpressionSyntax`, the resolved `IMethodSymbol`, the bound argument list, and a flag distinguishing operator calls from the terminator. The ordering and per-call shape is the part most exposed to T13's branched-chain decision (see Open questions).
- `Lambdas` — `ImmutableArray<LambdaExpressionSyntax>` of every lambda argument appearing inside the captured chain, in source order. The rewriter consumes these as opaque syntax; it does not re-bind them.
- `Captures` — `ImmutableArray<CapturedVariable>` covering every outer-scope local/parameter read inside the chain (including the bodies of `Lambdas`). Computed by a `ChainCaptureWalker` analogous to `QuotationCaptureWalker`. Each `CapturedVariable` carries `Name`, `Type` (`ITypeSymbol`), and `Location`.
- `Diagnostics` — `IDiagnosticReporter` mirroring `BindingDiagnosticReporter` from `[Quotable]`. Used both for "I cannot translate this" diagnostics and as the silent-decline signal: returning `null` with zero reported diagnostics triggers T11 silent fallback; returning `null` with at least one reported diagnostic is a hard error path.
- `GetFreshName(string hint)` — name generator seeded from a per-call-site counter so that wrapped-substitution mode (T10) gets stable, non-colliding identifiers without the rewriter needing access to the enclosing symbol table.

The type is sealed, immutable, and constructed only by the compiler. Analyzer authors never instantiate it; tests use an internal factory exposed via `InternalsVisibleTo`.

## Edge cases / risks

- **Bound-tree re-entrancy.** `SemanticModel.AnalyzeDataFlow` on the chain re-enters the binder for the expression currently being bound and stack-overflows; the analogous case is documented for `[Quotable]`. `ChainRewriteContext` must compute captures via the internal walker before construction, and the public contract for `SemanticModel` must warn rewriter authors against re-querying the chain itself.
- **Operator-agnostic shape.** T06 declares the chain operator-agnostic: any method returning the marked queryable type is part of the chain. The `ChainCall` shape must therefore not assume LINQ method names, generic-arity, or extension-vs-instance form. Any field with a `LinqMethod` flavor is wrong.
- **Captures inside chain lambdas.** `Where(x => x.Id == localId)` — `localId` is captured by the lambda, not by the chain call itself. The walker must descend into bound-lambda bodies attached to chain-call arguments, otherwise wrapped-mode substitution (T10) emits unresolved identifiers.
- **Fresh-name collisions in wrapped mode.** `GetFreshName` must be deterministic across a single rewriter invocation but distinct across concurrent compilations sharing the same rewriter instance. A per-context counter (seeded by a per-call-site value) avoids the `static int counter` trap that breaks concurrent builds.
- **Diagnostic descriptor identity.** Rewriter-supplied diagnostics need reference-identical `DiagnosticDescriptor` instances across reporter calls so downstream suppression and equality work — the `[Quotable]` design uses a static `ConcurrentDictionary<(id, severity, rewriterName), DiagnosticDescriptor>` cache. Mirror that here.

## Open questions

- **T13 (branched chains) determines the top-level shape of `ChainCalls`.** Three live options:
  - *per-terminator*: the context is constructed once per terminated path; `ChainCalls` stays a flat array; a chain that fans out invokes the rewriter N times with N overlapping contexts. Easiest to ship but means the rewriter cannot dedupe shared prefixes.
  - *whole-DAG*: `ChainCalls` becomes a tree/DAG shape with explicit fan-out nodes; the rewriter sees the shared prefix once. Larger API surface, requires the substitution layer (T10) to thread multiple substitution sites in one shot.
  - *linear-only v1*: any branching falls back per T11; `ChainCalls` stays flat and we revisit shape post-v1.
  This is the single biggest design decision pending for this task; defer the final type signature until T13 resolves.
- **T12 (`IAsyncEnumerable<T>`) decides what may appear inside `ChainCalls`.** If chains rooted at a marked type that also implements `IAsyncEnumerable<T>` are in scope, the array may include calls whose return type widens to `IAsyncEnumerable<T>` and terminators that bind via `GetAsyncEnumerator`/`await foreach`. Either the per-call type information must be uniform with `IQueryable<T>` chains, or `ChainRewriteContext` needs a flag (or a second context type) distinguishing them. Unresolved.
- **Should `ChainRewriteContext` expose a method-symbol view of the chain, or only the syntax?** `QuotationContext` exposes only syntax + `SemanticModel`; for queryable chains the rewriter almost always needs the resolved overload of every operator. Current draft exposes `IMethodSymbol` per `ChainCall`; reconsider if it overcommits to the binder's overload resolution result.
