# T18 — Benchmarks

Ship benchmarks measuring:

- Compile-time overhead per chain.
- Runtime per-site overhead vs today's `IQueryable` evaluation.

Cite — `v1-deliverables`: *"Benchmarks — compile-time overhead per chain + runtime overhead per call site vs runtime IQueryable"*.

## Outcome

Two BenchmarkDotNet suites land in the repo, both driven by the T15 Postgres sample as the system under test:

- **Compile-time suite (`CompileTimeQueryableBindBenchmarks`).** Drives `AdhocWorkspace` → `Compilation.GetCompilationAsync().Result.EmitToStream()` over a synthetic source that contains N chains rooted at a `[CompileTimeQueryable]`-marked provider. Reports three axes per N: (a) flag-off baseline (binder must short-circuit at T06 with zero per-chain cost — must overlap the no-feature compiler within noise), (b) flag-on with no chains detected, (c) flag-on with N chains rewritten. Marginal per-chain cost is the slope of (c) over multiple N values; the (c) − (b) gap at a single N divided by N gives the average and is the cruder fallback. The slope is what regresses when T06/T07/T09 add cost.
- **Runtime suite (`CompileTimeQueryableRuntimeBenchmarks`).** Compiles the same chain twice — once with `/features:compile-time-queryable` (emitting the Dapper-style materializer call) and once without (running through the provider's runtime `IQueryable` plumbing) — loads both assemblies in-process via collectible `AssemblyLoadContext` (netcore-only API, see TFM note in *Files to touch*), and benchmarks per-call wall-clock + allocations on a stubbed `IDbConnection` that returns a fixed `DataReader` over an in-memory row set. No real DB, no network.

The deliverable is two `[Benchmark]`-decorated classes plus enough documentation in their XML doc comments to explain what each axis means. Wiring into CI (perf-gating) is explicitly *not* part of v1 — manual `dotnet run -c Release --project IdeCoreBenchmarks` invocation is the contract, matching how `SwitchStatementBenchmarks` and `IncrementalSourceGeneratorBenchmarks` ship today.

## Files to touch

The existing compiler-side BDN harness is `src/Tools/IdeCoreBenchmarks/` — there is no `src/Tools/PerformanceTest/` or `src/Compilers/CSharp/Test/Tools/Perf/`. Verified by `grep -l BenchmarkDotNet src/**/*.csproj`: the only non-Razor hits are `IdeCoreBenchmarks.csproj` and `IdeBenchmarks.csproj`. `IdeCoreBenchmarks` is the right home — it already references `Microsoft.CodeAnalysis.CSharp`, `Microsoft.CodeAnalysis.CSharp.Test.Utilities`, and `Microsoft.CodeAnalysis.Compiler.Test.Resources`, which is the dependency set both suites need.

- `src/Tools/IdeCoreBenchmarks/CompileTimeQueryableBindBenchmarks.cs` (new) — mirror `SwitchStatementBenchmarks.cs`: `[MemoryDiagnoser]`, `[SimpleJob(RunStrategy.Throughput, invocationCount: 1)]`, a `[Params(1, 10, 100, 1000)]` over chain count, three `[Benchmark]` methods (`FlagOff`, `FlagOnNoChains`, `FlagOnNChains`) built from a shared `CreateSourceFile(int chainCount, bool emitChains)` template. The flag is set via `CSharpParseOptions.Default.WithFeatures(new[] { new KeyValuePair<string, string>("compile-time-queryable", "true") })` (matches the T04 wiring).
- `src/Tools/IdeCoreBenchmarks/CompileTimeQueryableRuntimeBenchmarks.cs` (new) — `[GlobalSetup]` drives `CSharpCompilation.EmitToStream()` twice against the T15 sample source (flag-on `CSharpParseOptions` for one, flag-off for the other), loads both byte arrays into separate collectible `AssemblyLoadContext` instances via `LoadFromStream`, resolves the entry method via reflection on first iteration, then runs `[Benchmark]` methods `EmittedDapperCall` and `RuntimeIQueryable` over a pre-built fake reader. Use `[IterationSetup]` to reset the reader cursor. Drive emit through `CSharpCompilation` directly, *not* `Microsoft.CodeAnalysis.CSharp.CommandLine.Program.Main` — the public entry routes through `BuildClient`, which can dispatch to the VBCSCompiler server (`Csc` itself is `internal sealed`), and that is the wrong path for an in-process benchmark.
- `src/Tools/IdeCoreBenchmarks/IdeCoreBenchmarks.csproj` — add `ProjectReference` to the T15 sample project (once it lands) so the runtime suite can pull in the rewriter assembly and the provider type. No new package references; `BenchmarkDotNet` is already there. Gate the runtime suite source file on `$(TargetFramework)` ≠ `net472` (via `<Compile Include="..." Condition="..." />`) or wrap the class in `#if NET` — collectible `AssemblyLoadContext` does not exist on the netfx target.
- `src/Tools/IdeCoreBenchmarks/Assets/CompileTimeQueryable/` (new directory) — small fixture source file containing the chain shape (`provider.Where(...).Select(...).ToList()`) used as input by both suites. Keeps the suite self-contained and avoids re-parsing T15's full sample on every iteration.

No changes to `IdeBenchmarks` (IDE-side, references `EditorFeatures`, wrong dependency layer for a compiler-only feature). No changes to `src/Test/Perf/` (legacy non-BDN harness driven by `tests/` shell scripts).

## Dependencies

- **Hard-blocks on T01–T11.** All eleven foundation tasks must produce a working flag-on compilation path: T01 (attribute well-known type), T02 (`IQueryableChainRewriter`), T03 (`ChainRewriteContext`), T04 (`/features:compile-time-queryable`), T05 (analyzer-driven discovery), T06 (chain detection), T07 (dataflow capture), T08 (terminators), T09 (rewriter invocation), T10 (source-text substitution), T11 (decline fallback). Without all eleven, the (c) axis of the compile-time suite is meaningless and the runtime suite cannot produce two different binaries.
- **Test subject:** T15 (canonical Postgres sample) is the rewriter under measurement. Both suites consume the T15 sample's provider type and rewriter assembly; the compile-time suite consumes them as inputs to a synthetic source, the runtime suite consumes the emitted call site. If T15 ships a fake `IDbConnection` for its own tests, the runtime suite reuses it; otherwise this task contributes the in-memory reader fixture.
- **Soft dependency on T16.** The compiler-test suite (T16) and the benchmark suite are independent deliverables, but the same `CSharpTestBase`-style helpers for declaring `[CompileTimeQueryable]` inline (until the dotnet/runtime PR merges, per T01 edge case) are reused. If T16 lands first, the helper code is already there to import.
- **Independent of T17, T19–T22.** Spec doc and out-of-scope statements have no interaction with benchmark code.

## Edge cases / risks

- **Flag-off regression is the load-bearing measurement.** The (a) axis exists specifically to prove the T04/T06 short-circuit holds: a build that does not opt into the feature pays *zero* per-chain cost relative to a binary-identical compiler that never knew about the feature. The benchmark must therefore run flag-off both with the feature-aware compiler binary and (via assembly hash check or `Compilation.GetDiagnostics()` byte-equivalence) verify the output matches a control. Without this anchor the suite cannot detect a regression where T06 walks every receiver-static-type even when the flag is off.
- **No DB I/O.** The runtime suite measures emitted-code overhead, not Postgres latency or `Npgsql` parsing. Fake `IDbConnection` / `IDataReader` returning a pre-built in-memory row buffer; if anything in the materializer path touches sockets, the suite is wrong. Caveat: an `IQueryable` provider that defers materialization to `Enumerable.Cast` or `Expression.Compile` will allocate `LambdaExpression` trees on every call — that allocation *is* the runtime-IQueryable overhead the comparison is meant to surface, so do not stub it out.
- **SQL-emit quality is out of scope.** The benchmark must not score "the rewriter produced shorter / faster SQL." That is a property of T15's rewriter author, not the compiler feature. Both suites measure host-side cost only: time spent in the binder (compile-time) and time spent constructing/invoking the materializer call (runtime). A reviewer should reject any column that reports query plan cost or row throughput.
- **Cold vs. warm rewriter assembly load.** The T05 discovery path loads the rewriter assembly on first chain in a compilation. The compile-time suite must distinguish cold load (first chain in the suite) from warm steady-state (chains 2…N). Use `[IterationSetup]` to construct a fresh `AdhocWorkspace` per iteration and separately tag the first chain's cost, or report cold and warm as distinct `[Params]`. Without this, the per-chain slope is polluted by a one-shot constant.
- **Variance on shared CI.** `IdeCoreBenchmarks` is not currently CI-gated; running it on shared hardware produces noisy results. Document a target environment (release build, `DOTNET_TieredCompilation=0`, `Server GC`, single-process, no other workload) in the suite's XML doc comment. Do not encode a numeric pass/fail threshold in code — record absolute medians + ratios in the suite output and leave the human-judged "is this regressed" to the PR reviewer, matching how `SwitchStatementBenchmarks` operates.
- **`AssemblyLoadContext` lifetime in the runtime suite.** Loading two `compile-time-queryable`-flag-on and flag-off DLLs into the default ALC will collide on assembly identity. Each must load into its own collectible ALC; `[GlobalCleanup]` must unload both. Forgetting this leaks one assembly per benchmark run and falsifies `[MemoryDiagnoser]` output.
- **`CSharpTestBase` is a unit-test type.** The IDE precedent (`SwitchStatementBenchmarks`) uses `AdhocWorkspace` + `MetadataReference.CreateFromFile(typeof(object).Assembly.Location)` directly, not `CSharpTestBase`. Follow that pattern — `CSharpTestBase` carries xUnit machinery (`Fact`, `Theory`) and assertion plumbing that does not belong in a BDN process. Both suites should drive `CSharpCompilation` directly; the command-line `Program.Main` entry is not viable (server dispatch via `BuildClient`).

## Open questions

- **T12 (`IAsyncEnumerable<T>` scope) expands the matrix.** If async chains stay captured, the compile-time suite needs an additional `[Params]` axis for sync vs. async terminators (T08 covers `ToListAsync` / `await` / `await foreach`), and the runtime suite needs an `IAsyncEnumerable<T>`-yielding fake reader to compare against an async `IQueryable` provider. Surface only — do not bake the async columns in until T12 resolves.
- **T13 (branched chains) expands the matrix again.** If T13 picks `whole-DAG`, the compile-time suite needs a branched-chain fixture distinct from the linear one (different per-chain slope expected — the rewriter sees a tree, not a list). If T13 picks `per-terminator`, the linear suite already covers it. If T13 picks `linear-only v1`, the suite must include a branched fixture that exercises the T11 fallback path so regressions in the fallback-detection logic are visible. The choice changes the suite's class count but not its file layout.
- **Inlined vs. wrapped substitution.** T10 may emit either an inline substitution or a synthesized static-method wrap. The runtime suite measures the call site as emitted; if the inline-vs-wrap choice depends on input shape, the suite may need both variants as separate `[Benchmark]` methods to surface the per-call dispatch cost (direct call vs. extra static-method indirection). Decide once T10 lands.
