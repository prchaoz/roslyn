# T17 — Language design spec

Ship a language design spec under `docs/specs/`.

Cite — `v1-deliverables`: *"Language design spec under docs/specs/"*.

## Outcome

A speclet for the *compile-time queryable chain rewriter* feature, drafted as a prototype-phase document and promoted to a final spec once the implementation stabilizes. The document is the canonical reference for binder behavior, runtime fallback policy, analyzer-loading model, and the feature's intentional scope — it must read accurately enough that a contributor unfamiliar with the goal can implement T01–T11 from it.

Required sections, in order:

1. **Motivation.** Why a separate mechanism from `Expression<T>` and from `[Quotable]`: provider-pluggable chain rewriting (Postgres-style table → SQL + materializer), zero runtime IQueryable allocation when the rewriter succeeds, and a fallback path that preserves source-compatibility. State explicitly that this is **not** shared plumbing with `[Quotable]` (T20).
2. **Syntax and types added.** No new syntax. New surface area: `[CompileTimeQueryable(string rewriterAssemblyQualifiedName)]` attribute (T01), `IQueryableChainRewriter` interface with `string? Rewrite(ChainRewriteContext)` (T02), `ChainRewriteContext` shape (T03). All three sit under `Microsoft.CodeAnalysis.CSharp.*` with `[Experimental("RSEXPERIMENTAL00N")]`.
3. **Binder behavior.** How `BindInvocationExpression` (and `BindAwaitExpression` / `BindForEachExpression` for async + foreach terminators) detect that the receiver static type carries `[CompileTimeQueryable]`, walk the local-dataflow chain (T06–T07), classify a terminator (T08), and dispatch to the rewriter (T09). Specify that detection runs at binding time, not lowering, so semantic-model consumers see the substituted form.
4. **Terminator semantics.** Enumerate the four terminator families from the interview (materializers; aggregates / single-result operators; `foreach`; `await`) and the operator-shape rule (chain = any call returning the same marked queryable type). State the rule that operator identity does not matter to the compiler — the rewriter decides what it translates. Spell out async terminators only after T12 resolves.
5. **Rewriter contract.** Inputs (`ChainRewriteContext` shape per T03), outputs (`string?` per T02), declination semantics (null + diagnostics → fallback per T11), required statelessness across chain sites, allowed output shapes (single expression that binds at the chain site, IIFE for multi-statement, async lambda once T12 is resolved). Cross-reference T08's terminator list and T13's chain-shape decision.
6. **Substitution model.** Inline substitution by default (`SyntaxFactory.ParseExpression` + `BindExpression` at the chain site, captures resolve lexically), wrapped-static-method fallback when inline binding fails. Specify how the wrapper's typed capture record is synthesized (per-site `SynthesizedClosureEnvironment`-shaped container, strongly typed field per captured local) and that the substituted form must satisfy `ClassifyImplicitConversionFromExpression` against the chain's destination type before the identity-wrap is emitted. Mirror Quotable's substitution discussion but explicitly state where this feature diverges (chain receiver vs. lambda operand).
7. **Runtime fallback policy.** When the rewriter declines, returns invalid source, throws, or is missing entirely, the chain runs at runtime through the marked type's normal `IQueryable` / `IEnumerable` plumbing (cite `decline-behavior`). Specify that fallback is **silent by default** — declination produces no warning at the chain site — and explain the trade-off vs. a noisy mode. Document the planned `RSEXPERIMENTAL00N`-suppressible warning hook (T11) for IDE-discoverability.
8. **Analyzer-loading model.** Rewriter implementations ship as analyzers; the AQN in `[CompileTimeQueryable]` is resolved against the loaded analyzer set via `CommonCompiler.PopulateLanguageSpecificRuntimeState` (T05), populating a per-compilation `QueryableChainRewriterRegistry`. Mirror Quotable's plumbing. **Security implication** must be called out explicitly: any analyzer in the project can register itself as a rewriter for any marked type the project references; rewriters execute user code in the compiler process at build time. Cross-reference `dotnet build` / NuGet trust posture — same risk model as source generators and analyzers in general, but a new attack-surface entry worth naming in the spec.
9. **Feature-flag gating.** `/features:compile-time-queryable` (T04). Without the flag, `[CompileTimeQueryable]` is ignored and chains bind as normal IQueryable / IEnumerable invocations. State that the flag is required even after the prototype merges to main, and that final acceptance ties the feature to a `/langversion` switch and removes the flag (per `Developing a Language Feature.md`).
10. **Out of scope.** Verbatim from the interview's `out-of-scope` set: VB.NET (T19), shared plumbing with `[Quotable]` (T20), IDE refactorings / IntelliSense / "view emitted code" UI (T21), reimplementing `Expression<T>` via this mechanism (T22). One sentence each, citing the interview answer.
11. **Future considerations.** BCL rewriter-author helpers / generic visitor base classes (T14, pending), expanded async surface (T12, pending), branched-chain handling (T13, pending), eventual IDE quick-actions for "view what the rewriter produced", potential overlap with future interceptor-based prototypes. Each entry is one sentence pointing at the open question.

The spec also carries a one-paragraph **decision rationale** appendix recording why operator-agnostic was chosen over an operator-allowlist, why silent fallback was preferred over a warning, and why the discovery mechanism mirrors Quotable instead of inventing a parallel registration channel — all three are non-obvious choices from the interview and will be re-litigated otherwise.

## Dependencies

- **Hard depends on T12, T13, T14.** The spec sections covering async-chain scope (§4 terminator semantics, §11 future considerations), chain-shape (§3 binder behavior, §5 rewriter contract), and BCL helpers (§10 out-of-scope vs. §11 future considerations) must reflect a **decided** state, not the interview-state "I think about…" punt. Drafting the spec before T12–T14 resolve produces a document that codifies the wrong defaults; resolving T12–T14 after the spec ships produces a spec that drifts. **Final spec must wait** until all three open questions are answered.
- **Soft depends on T01–T11.** The prototype draft can start as soon as T01 (attribute), T02 (interface), and T04 (feature flag) have stable shapes, since those define the public surface area the spec describes. Sections on binder behavior (§3), substitution model (§6), and analyzer loading (§8) need T05–T11 stable so the spec describes *what the compiler actually does*, not a wish.
- **Depends on T15 (sample) and T16 (compiler tests)** only for citation — the spec's "validating examples" links to the canonical Postgres sample (T15) and the test sweep (T16). The spec does not block on either.
- **Blocks T18 (benchmarks)** only in the sense that the benchmark methodology section should cite the spec's terminator inventory; T18 can otherwise proceed independently.
- **No direct dependency on T19–T22.** Out-of-scope tasks are documented in §10 by citation, not by reading their contents.

## Edge cases / risks

- **Spec written too early → drift.** The biggest failure mode: drafting §3 (binder behavior) and §6 (substitution) before T06–T10 stabilize produces text that contradicts the implementation within weeks. Mitigation: the prototype draft explicitly marks each section with the T-number it tracks, and a section is only marked "stable" once its T-tasks are merged. Sections describing T12/T13/T14 stay marked `*(unresolved — pending T1N)*` until those tasks resolve, and the final spec promotion gate is "no unresolved markers remain."
- **Analyzer-discovery security implication left implicit.** The Quotable spec sibling discusses analyzer discovery as a mechanics detail; this spec must escalate it to a numbered subsection (§8) because the chain-rewriter executes user code in the compiler process *and* directly emits source text spliced into the user's program. Missing this section misses a real attack-surface entry and will be a reviewer-stop in language design review.
- **Decline-fallback IDE-discoverability glossed over.** Silent fallback (per `decline-behavior`) is the right default but a real usability hazard: a developer changes a chain operator, the rewriter silently declines, the code now runs through the legacy IQueryable path, and runtime perf regresses without any compile-time signal. §7 must walk through this scenario and document the planned diagnostic-suppression hook for noisy-mode opt-in (cross-ref T11). Omitting this discussion means future bug reports against "why did my chain stop being inlined?" have no spec text to point at.
- **Test-matrix coverage of unresolved branches.** Once T12 (`IAsyncEnumerable<T>`), T13 (branched DAGs), and T14 (BCL helpers) resolve, the spec's terminator inventory (§4), chain-shape rule (§3), and rewriter-author API surface (§5 + §11) all change. §11 must list, per decision, which T16 test buckets the resolved branch needs to add — otherwise the test sweep ships with coverage holes in exactly the recently-stabilized areas. Coordinate with T16's enriched task file once T12–T14 close.
- **Final-spec location naming.** `CSharpNN` placeholder is fine for the prototype draft, but the promotion PR must pick the right `NN` (the language version at which the feature is accepted). Mis-naming the directory at promotion is a rename-everything cleanup.
- **Localization side effect.** None expected. Spec text is English-only under `docs/`; no `.resx` regen.
- **Interaction with `Better Betterness.md` exemplar shape.** That exemplar reads as proposed numbered insertions into the C# specification because the C# 6 feature modified overload resolution. This feature does **not** modify overload resolution or any other numbered specification clause — the binder dispatches *after* overload resolution picks a method that returns a marked type. The spec should adopt the exemplar's numbered-subsection style only where it genuinely describes spec-language insertions (none, for v1); the rest reads as a feature-overview document. Do not over-fit to the exemplar's form.

## Open questions

- **T12 outcome dictates §4 and §11.** If async chains are in scope, §4 documents `await`-on-chain and `await foreach` as first-class terminators with allowed `async`-lambda rewriter outputs; §11 has nothing to say. If async is deferred, §4 documents only sync terminators and §11 carries a "post-v1 expansion" entry. Spec authoring is blocked on T12 resolution before §4 final-form.
- **T13 outcome dictates §3 binder text and §5 contract.** Per-terminator linear chains → §3 describes a linear walk and §5's `ChainRewriteContext` is a list; whole-DAG → §3 describes the DAG-projection step and §5's context is tree-shaped; linear-only v1 → §3 documents branching as an immediate fallback trigger and §5 unchanged. The rewriter contract section (§5) cannot reach final form without T13's decision.
- **T14 outcome dictates §10 vs. §11 placement.** If BCL helpers are in scope, §11 carries a "shipping `IRewriterBase` and visitor utilities" entry and the namespace home for `IQueryableChainRewriter` may move from `Microsoft.CodeAnalysis.CSharp.*` to `System.Runtime.CompilerServices` so BCL helpers can reference it without a Roslyn dependency (cross-ref T02). If out of scope, §10 carries a verbatim citation from `out-of-scope` and `IQueryableChainRewriter` stays Roslyn-side.
- **Namespace name** — `Microsoft.CodeAnalysis.CSharp.QueryableChain` vs. `…CompileTimeQueryable`. Decided here in §2 once T01/T02/T03 agree on a single namespace; surface the choice in the spec so downstream NuGet authors have a stable target.
- **Final spec directory `NN` value.** Picked at promotion time. Until then the file lives only at `docs/features/compile-time-queryable.md`. Surface this as an open item so the promotion PR doesn't have to re-derive it.
- **Quotable-spec sibling reference.** The Quotable feature's plan and facts live in a sibling project outside this repository, so the spec cannot cross-link to them. Either inline the relevant Quotable contract discussion in §1 (motivation, "why not Quotable") or wait for the Quotable spec to land in `docs/features/` / `docs/specs/` and cross-link then. Defer to spec author at draft time.
