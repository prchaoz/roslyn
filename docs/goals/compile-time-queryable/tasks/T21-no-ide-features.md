# T21 — *(out of scope)* No IDE features

Do not ship IDE refactorings, IntelliSense extensions, or "view emitted code" UI.

Cite — `out-of-scope`: *"IDE refactorings / IntelliSense extensions / 'view emitted code' UI"*.

## Outcome

The feature ships strictly in the C# compiler. The IDE picks up the new symbols, attribute, and diagnostics for free through the shared `Microsoft.CodeAnalysis.CSharp` surface; nothing in this v1 adds, removes, or specializes IDE-layer behavior.

Concretely, what the IDE *does* get with no extra work:

- **IntelliSense across the chain is unchanged.** The rewriter is invoked after binding completes and before IL emit (T09); under T10's substitution model, no rewritten syntax trees are exposed to the IDE-facing `SemanticModel`, which keeps walking the pre-rewriter bound tree. Completion, signature help, and hover on `table.Where(x => x.|)` show members of `x`'s type just as they would for a runtime `IQueryable<T>` chain. If T10's chosen mechanism does expose new syntax to the public `SemanticModel`, this Outcome needs to be revisited — flag it in spec review.
- **Error squiggles flow naturally.** Missing-rewriter (T05), declined-with-error (T09/T11), and invalid-output (T10) all surface as ordinary compiler diagnostics anchored on the chain's terminator call site. The IDE shows them through the same path as `CS0103`/`CS1061`; no new diagnostic UI is needed.
- **`[Experimental]` warnings on `IQueryableChainRewriter` / `ChainRewriteContext` flow through the existing experimental-attribute lightbulb.** Suppression, code-fix, and squiggle behavior all come from the analyzer that ships with the `[Experimental]` attribute today — no T21 work.

What the IDE explicitly does *not* get in v1:

- No "view emitted code" pane for the substituted source. The rewriter's output is observable through compiler tests (T16) and binlogs; there is no Visual Studio / VS Code surface that renders it.
- No refactoring to "extract this chain to a `[CompileTimeQueryable]` provider" or to generate a rewriter skeleton from a sample chain.
- No quick action to toggle the `compile-time-queryable` feature flag (T04) on or off at the chain site, the file, or the project. Users edit `csproj` or `Features` by hand, same as every other `LanguageVersion` / preview flag.
- No IntelliSense extension that previews what the rewriter will emit while typing — completion remains driven by the pre-rewriter `SemanticModel`.

## Files to touch

None in `src/EditorFeatures/`, `src/LanguageServer/`, `src/Features/`, or `src/VisualStudio/`. The feature is implemented entirely in `src/Compilers/CSharp/Portable/` and `src/Compilers/Core/Portable/`. Any PR for this feature whose diff touches the IDE trees is in violation of T21 and should be rejected at review time.

The single permitted IDE-adjacent touch is the autogenerated `src/Tools/SemanticSearch/ReferenceAssemblies/Apis/Microsoft.CodeAnalysis.CSharp.txt` diff, which the build produces from T02/T03's public API additions; it lives under `src/Tools/` and is not an IDE-layer change.

## Dependencies

- **Guards all of T01–T18.** Every implementation task lives in the compiler. T21 is the standing review rule that none of them grow IDE-layer scope creep.
- **T17 (spec)** must call this scope boundary out explicitly so external readers do not file requests for IDE refactorings or "view emitted code" UI against v1.
- **T16 (compiler tests)** is the substitute observability surface — anyone wanting to inspect rewriter output reads compiler tests rather than opening an IDE pane. Whether T16 includes snapshot/golden coverage of emitted source is a T16 decision, not a T21 requirement; T21 only records that the absence of an IDE pane is acceptable because the compiler-test surface exists.

## Edge cases / risks

- **F12 Go-to-Definition on a chain call.** The substituted code is synthesized after binding, before IL emit; the binding the IDE consults predates substitution. F12 on `table.Where(...)` therefore navigates to the original `Where` method declaration (the one on the marked queryable type), not to the rewriter's translated SQL fragment or the synthesized wrapper method (T10). This is the expected behavior — F12 follows the binder, and the binder sees the original chain — but it is worth nailing down in T17 so users do not interpret the absence of "F12 into the rewriter" as a bug.
- **Semantic classification of chain calls.** Method-name coloring, parameter coloring, and reference highlighting on a chain call run off the pre-rewriter `SemanticModel`. Chain calls render with the same colors as any other method call; there is no special "this call will be rewritten" tint. Adding one would be IDE work and is out of scope.
- **Debugger stepping into the synthesized wrapped method (T10).** When T10 chooses wrapped-mode substitution, the chain becomes a call to a compiler-synthesized method. Whatever debugger-attribute set T10 chooses (compiler-generated marker plus whatever step-through behavior the team picks, mirroring how lifted-lambda methods are handled today) determines how the debugger surfaces the wrapper. T21's only commitment is that no IDE-side debugger work is in scope to compensate if T10's defaults feel wrong — that is a T10 decision and a v2 follow-up if revisited.
- **Quick info / hover on the chain terminator.** Hovers show the original terminator's signature (`ToList<T>()` etc.), not any rewriter-emitted variant. If the rewriter rewrites `ToList` to a direct materialization call, hover still describes `ToList`. Acceptable for v1; revisit only if user reports show confusion.
- **Code lens / IntelliCode-style suggestions** sitting on top of the compiler symbol APIs see the chain as-is. No new symbols or analyzers are surfaced to them; if they highlight chain operators, they highlight the originals.

## Open questions

None. Any IDE-layer follow-up is a valid v2 candidate to be filed on top of the shipped v1; T21's job is only to keep all such work off the v1 plate.
