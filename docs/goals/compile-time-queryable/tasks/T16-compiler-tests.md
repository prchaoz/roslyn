# T16 — Compiler tests

Ship compiler tests covering:

- Binding at the chain site.
- Emit for inlined substitution.
- Emit for wrapped-method substitution.
- Error scenarios: missing rewriter, decline, invalid rewriter output.

Cite — `v1-deliverables`: *"Compiler tests — binding, emit (inlined + wrapped), error scenarios (missing rewriter, decline, invalid output)"*.

## Outcome

Three test classes, all deriving from `CSharpTestBase`, exercising the feature end-to-end against the implementation landed by T01-T11. Each suite uses raw string literals (`"""…"""`) for sources per AGENTS.md, asserts diagnostics through `VerifyDiagnostics` / `VerifyEmitDiagnostics`, and binds the test rewriter via the same registry-population path the production CSC pipeline uses (T05), not via a back-door test seam — so any drift in the analyzer-discovery wiring shows up as a test failure rather than a silently green run.

**`CompileTimeQueryableBindingTests`** (chain detection + dataflow capture + terminator recognition). Source under `src/Compilers/CSharp/Test/Semantic/Semantics/`. Covers, at minimum: receiver-static-type recognition (T06) for both root and continued calls; operator-agnostic continuation (no `WellKnownMember` table); reduced extension-method calls; generic-instantiation cross-overs that keep `OriginalDefinition` identity (`Postgres9Table<User>.Select(...) -> Postgres9Table<string>`); conditional-access (`?.`) receivers; query-syntax (`from x in t where … select …`) chains; intra-method local-variable dataflow capture (T07) including reassignment (`q = q.Where(...);`) and pattern/`out var` introductions; terminator recognition (T08) for materializers (`ToArray`/`ToList`/`ToDictionary`/`ToHashSet`), aggregates (`Single`/`First`/`Last`/`Count`/`Any`/`All`/`Sum`/`Min`/`Max`/`Average` and their `*OrDefault` siblings), `foreach`, and `await ToArrayAsync()`; escape-terminates-chain cases (return, field/argument escape); feature-flag-off semantics where every assertion above must be bit-identical to today's compiler (no `BoundCompileTimeQueryableChain` materialized, no diagnostics, no IL change). Each test uses an inline minimal `[CompileTimeQueryable("StubRewriter, TestRewriterAsm")]` provider class plus a single AQN-registered `StubRewriter` whose `Rewrite` is wired via a test-only `CSharpCompilation.SetRewriterRegistryFromAssemblies(...)` call mirroring what `CSharpCompiler.PopulateLanguageSpecificRuntimeState` does at the command line.

**`CompileTimeQueryableEmitTests`** (substitution emit). Source under `src/Compilers/CSharp/Test/Emit3/Semantics/` (where new emit suites for binder-driven rewrites have been landing — `LabeledBreakContinue*` lives in `Test/CSharp15/`, but that project is feature-version-gated, whereas `compile-time-queryable` is `/features:`-gated, which matches the `Emit3` siting pattern). Two arms:

1. *Inlined substitution* — the rewriter returns C# source text that binds cleanly at the call site; assert via `CompileAndVerify(...).VerifyIL("M", expected)` for at least three substitution shapes mirroring the `[Quotable]` IL matrix: (i) a parameter-free constant (`new MyProvider<Row>("SELECT …")`), (ii) a constant pulled from a captured local (lexical resolution against the call-site scope), (iii) a method-group reference (no `ldftn`/`Func` allocation). Pair each with `CompilationVerifier.VerifyDiagnostics()` and `comp.VerifyEmitDiagnostics()` so missing-but-legal scenarios still light up at review.
2. *Wrapped-method substitution* (T10's wrapped path, sibling to the inline path that fell out of the `[Quotable]` implementation). Assert the synthesized static method and capture-record-typed parameter via `CompileAndVerify(...).VerifyTypeIL("<CompileTimeQueryable>__Captures0", expected)` and the call-site IL via `VerifyIL`. The capture record's field order and name must be deterministic — see Risks below.

Cross-cutting: both arms must emit identical observable IL across consecutive runs on the same compilation (golden-test reproducibility), and the wrapped form's synthesized member names must be stable across `TestOptions.Regular` and `TestOptions.DebugExe`/`ReleaseExe`. Use `verify: Verification.Skipped` only when the substituted source intentionally produces unverifiable IL; default to `Verification.Passes`.

**`CompileTimeQueryableErrorTests`** (error scenarios). Source co-located with binding tests in `src/Compilers/CSharp/Test/Semantic/Semantics/`. One test per `ERR_Rewriter*` code reserved by T05 (`ERR_RewriterNotFound`, `ERR_RewriterWrongInterface`, `ERR_RewriterCtorThrew`, `ERR_RewriterAmbiguous`) and per T09/T10/T11 emission (`ERR_RewriterThrew`, `ERR_RewriterDeclined`, `ERR_RewriterReturnedInvalidSource` — three by analogy with `ERR_QuoterThrew`/`ERR_QuoterDeclined`/`ERR_QuoterReturnedInvalidSource`, exact codes owned by T09/T10/T11 not this task). Specifically:

- *Missing rewriter* — AQN points at a type that does not exist in any registered analyzer assembly. Expect one diagnostic per chain root (no global suppression), each at the chain's leading invocation `Location` (not at the `[CompileTimeQueryable]` attribute), with the AQN string echoed in the message via `WithArguments(...)`.
- *Decline with no diagnostic* — `StubRewriter.Rewrite` returns `null` without reporting through `ChainRewriteContext.Diagnostics`. Expect exactly one `ERR_RewriterDeclined` at the chain root. Sibling test: returns `null` *after* reporting at least one diagnostic — assert the reporter's diagnostic shows up and `ERR_RewriterDeclined` does *not* (T11's contract).
- *Syntactically invalid rewriter output* — `Rewrite` returns `"not-a-valid-expression $$$"`. Expect `ERR_RewriterReturnedInvalidSource` with the parse-error message concatenated; the call-site `Location` is the chain root, not an invisible synthesized location.
- *Type-mismatched but parseable output* — `Rewrite` returns `42` for a destination type of `IEnumerable<User>`. Expect `ERR_RewriterReturnedInvalidSource` with a type-mismatch detail (`ClassifyImplicitConversionFromExpression` non-existent path).
- *Rewriter throws* — `StubRewriter.Rewrite` throws `InvalidOperationException("rewriter bug")`. Expect `ERR_RewriterThrew` carrying the *inner* exception's message (unwrapped from `TargetInvocationException`).
- *Fallback survives error scope* — for codes that the spec marks as warnings-equivalent (none currently; placeholder), assert the chain falls back to runtime evaluation. Otherwise, assert that the chain root is bound as a `BoundBadExpression` so downstream diagnostics don't cascade.

- *Not touched* — `src/Compilers/CSharp/Test/CommandLine/CommandLineTests.cs` does not need a `compile-time-queryable` arm in T16: T05 owns command-line analyzer-loading coverage. T16 verifies semantics *given* a populated registry, not the population path itself.

Test compilations use `TestOptions.Regular.WithFeature("compile-time-queryable", "true")` (or its `WithFeatures` equivalent) to flip T04's flag; sibling flag-off tests use plain `TestOptions.Regular` and assert identical output to the no-feature baseline. The MEF/`[UseExportProvider]` attribute is *not* needed: T05 wires the registry through a settable internal property on `CSharpCompilation` (`SetRewriterRegistryFromAssemblies`), not through a MEF composition. `[UseExportProvider]` would only be required if a test loaded the IDE-side analyzer host, which is out of scope for these compiler tests (and explicitly out of scope per T21).

## Edge cases / risks

- **Test rewriters loaded as inline analyzers vs. reference projects.** A real `[CompileTimeQueryable]`-using NuGet ships an analyzer assembly under `analyzers/dotnet/cs/`; tests can't reasonably build and load that on every run. The `InlineRewriterAnalyzer` shim sidesteps the analyzer-driver entirely by calling `CSharpCompilation.SetRewriterRegistryFromAssemblies(ImmutableArray.Create(typeof(StubRewriter).Assembly))` directly. Risk: this exercises the registry's `Resolve(aqn)` path but skips `CSharpCompiler.PopulateLanguageSpecificRuntimeState`'s analyzer-walk → distinct-assembly → set-on-compilation pipeline. Mitigation: add at least one `CommandLineTests`-style integration test (sited *there*, by T05's owner, not in T16) that exercises the full `csc /analyzer:` → `SetRewriterRegistryFromAssemblies` path, and rely on T16 for everything downstream.
- **Test fakes for `IQueryableChainRewriter`.** Stub rewriters need to implement the experimental interface, so each fixture file needs `#pragma warning disable RSEXPERIMENTAL00N` at the top (T02 reserves the id). Forgetting the pragma fails CI's experimental-API gate.
- **Deterministic AQN matching across test runs.** `Assembly.GetName().Name` for the test assembly is `Microsoft.CodeAnalysis.CSharp.UnitTests` (or `Microsoft.CodeAnalysis.CSharp.Emit3.UnitTests`), which is stable, but tests must use that exact name in the `[CompileTimeQueryable("StubRewriter, <thisAssembly>")]` string. Computing the AQN as `typeof(StubRewriter).AssemblyQualifiedName` at runtime and embedding it via `string.Format` into the test source (rather than hard-coding the assembly portion) makes the tests robust to test-assembly-name changes.
- **Capture-record naming determinism for golden tests.** The wrapped-method path synthesizes a private capture record per call site; the type name pattern (e.g. `<CompileTimeQueryable>__Captures{counter}`) and field names must be deterministic across runs of the same compilation. T10 owns the naming scheme; T16's golden-IL assertions are the *enforcement* mechanism. If T10 picks a name derived from `SyntaxNode.Span` rather than a monotonic counter, golden tests need to use stable source positions (raw string literals with explicit indentation are sufficient).
- **Emit-test stability across compiler updates.** IL goldens routinely break on unrelated lowering changes (Roslyn's IL printing differs between sequence-point placement updates, `nop` insertion changes, etc.). Mitigation: scope `VerifyIL` matches to the substitution-relevant instructions only (use `// sequence point: ...`-style elision per `LabeledBreakContinueEmitTests`'s pattern), and pair each `VerifyIL` with a `CompileAndVerify(..., expectedOutput: "...")` runtime check so a regression in the goldens that is actually correct lowering is easy to confirm by re-running with the new IL pasted in.
- **`TargetFramework` and runtime gating.** Async-terminator tests (`await ToArrayAsync()`) need `TargetFramework.Net100` (or whichever ships `IAsyncEnumerable<T>` extension surface). Materializer/aggregate tests using LINQ surface can stay on the default `TargetFramework.Standard`. Mirror `LabeledBreakContinueEmitTests`'s `IncludeExpectedOutput` helper for runtime-conditional `expectedOutput` so non-CoreCLR CI legs (e.g. net472) don't try to execute Net10-only assemblies.
- **Flag-off bit-identicality.** Asserting "no IL change with the flag off" requires comparing emitted IL byte-for-byte against the baseline. A practical proxy is to assert that the *same source* with the flag off produces `BoundCall` nodes only (no `BoundCompileTimeQueryableChain` materialized) via `GetSyntheticDiagnostics` and a bound-tree dump. Full byte-level IL diffing is brittle and not worth the maintenance cost; assert diagnostic-set equality and a single representative `VerifyIL` instead.
- **Cascading diagnostics on error-scenario tests.** If a rewriter fails (`ERR_RewriterReturnedInvalidSource`), the chain root should bind to a `BoundBadExpression` so terminator-side and downstream-conversion diagnostics don't pile on. Tests must assert exactly one diagnostic per failure (use `.Single()` per AGENTS.md) — a cascade indicates a missing `HasErrors` propagation in T09/T10/T11.
- **MEF / `[UseExportProvider]`.** Not needed (see Files to touch). Calling out explicitly because the task instructions raised the question; if a future task adds MEF-loaded discovery (e.g. T21's IDE work), the IDE-side tests will need `[UseExportProvider]` but the compiler suites here do not.

## Open questions

- **T12 (`IAsyncEnumerable<T>` scope).** If async chains stay captured, `CompileTimeQueryableBindingTests` gains a terminator-recognition arm for `await foreach` and `WhereAwait`; `CompileTimeQueryableEmitTests` gains an async-method-rewrite arm. If async chains are deferred, those arms become "feature-flag rejects chain at IAsyncEnumerable transition" tests. Either way the file layout is unchanged.
- **T13 (branched chains).** If T13 picks per-terminator linear chains, `CompileTimeQueryableBindingTests` adds a "shared prefix duplicated across terminators" assertion. If T13 picks whole-DAG, the assertion grows to "rewriter sees a tree-shaped `ChainRewriteContext`" — and `CompileTimeQueryableErrorTests` gains a "branched chain with one declining branch" matrix entry. Surfaces only.
- **T14 (BCL helpers scope).** If BCL ships helper materializer/aggregate base classes, the test fixtures may need to reference them (a test-only polyfill assembly if BCL hasn't shipped yet), mirroring T01's inline-declaration pattern for the attribute. Pure scope question; no mechanics change.
