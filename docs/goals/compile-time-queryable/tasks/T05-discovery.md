# T05 — Rewriter discovery & loading

Load rewriter implementations via the analyzer extension mechanism. Match the assembly-qualified name in `[CompileTimeQueryable]` against loaded types. Missing or unresolvable rewriter → error diagnostic at each dependent call site.

Cite — `discovery`: *"(a) Analyzer mechanism + assembly-qualified name in the attribute (mirrors `[Quotable]`)"*.

## Outcome

A per-`CSharpCompilation` `RewriterRegistry` keyed by the assembly-qualified name (AQN) string supplied to `[CompileTimeQueryable]`. Populated once per compile, after `ResolveAnalyzersFromArguments` and before the first `GetDiagnostics` call (this is the same staging window the `[Quotable]` plan validates as the only place where the loaded analyzer instances and the not-yet-bound compilation are both in hand — see `/Users/sergei.lipin/custom-expression-trees/plan.md` Step 3). The registry caches `(success, IQueryableChainRewriter instance)` or `(failure, reason)` for each AQN; a single resolution failure produces one diagnostic per dependent call site (T09 reports it at the chain root). The registry is runtime-only state — not part of compilation identity — so it is held in an internal settable property rather than threaded through the immutable `With*` rebuild pattern.

AQN matching is bracket-aware: split the AQN on the *outermost* comma (the one that separates `TypeName` from `AssemblyName, Version=...`). Generic AQNs (`Foo\`1[[Bar, AssemblyA, ...]], AssemblyB, ...`) have inner commas inside `[[...]]` that a naive `IndexOf(',')` would split on incorrectly. Filter the registered analyzer assemblies by `assembly.GetName().Name` against the AQN's assembly portion, then call `Assembly.GetType(typeName, throwOnError: false)` with the type's full name only — `Assembly.GetType` rejects the full AQN form. Verify the resolved type implements `IQueryableChainRewriter`, then instantiate via `Activator.CreateInstance` and unwrap any `TargetInvocationException` to surface the user's actual constructor failure.

Error reporting policy: discovery itself reports nothing. The registry returns a typed failure object (`NotFound`, `WrongInterface`, `CtorThrew`, with payload), and T09's invocation site translates that into a `CS9xxx`-class diagnostic at the chain root (not at the `[CompileTimeQueryable]` attribute) so the user sees the error at the call site they wrote. Multiple chains referencing the same broken AQN each get their own diagnostic — there is no global "fail once" suppression.

## Dependencies

- **Requires T01** — the `[CompileTimeQueryable]` attribute and its well-known type/member registration must exist before the registry has anything to match against.
- **Requires T02** — `IQueryableChainRewriter` must be defined before the registry can interface-check resolved types.
- **Gated by T04** — `CSharpCompilation.IsCompileTimeQueryableEnabled` (the `/features:compile-time-queryable` query) wraps the `PopulateLanguageSpecificRuntimeState` override body: with the flag off, the registry stays empty and the discovery work is skipped, leaving build performance untouched for non-consumers.
- **Feeds T09** — the rewriter invocation step calls `compilation.RewriterRegistry.Resolve(aqn)` per chain root and translates the result (or failure) into either rewriter dispatch or the diagnostic codes reserved here.
- **Orthogonal to T20** — explicitly does NOT share plumbing with `[Quotable]`'s `QuoterRegistry`; the parallel implementation is intentional even though the design pattern is identical.

## Edge cases / risks

- **Duplicate AQNs across analyzers.** Two analyzer packages can declare a rewriter type with the same name in assemblies sharing a simple name. The lookup must enumerate *all* matching assemblies before failing, and on ambiguous match emit a distinct `ERR_RewriterAmbiguous` (not `ERR_RewriterNotFound`) so the user knows to disambiguate via assembly version/key in the AQN.
- **Assembly load failures vs missing types.** `AnalyzerFileReference.GetAssembly()` (`AnalyzerFileReference.cs:674`) raises via the existing analyzer-load diagnostic path well before T05 runs — those failures surface as `CS800x`-class diagnostics, not as ours. Inside the registry, a *successfully loaded* analyzer assembly that nevertheless lacks the named type is a `NotFound`, distinct from an assembly that fails to load.
- **Null `analyzer.GetType().Assembly`.** Defensive: skip null entries (`if (assembly is null) continue;`) — the `[Quotable]` PR review caught this as a real edge case for source-generated stub analyzers.
- **`Activator.CreateInstance` exception wrapping.** User rewriter constructors that throw produce `TargetInvocationException`; always unwrap to `InnerException` so `ERR_RewriterCtorThrew` carries the user's actual message rather than "Exception has been thrown by the target of an invocation".
- **Analyzer hot-reload (IDE scenario).** A new `CSharpCompilation` is produced on every meaningful change, and the registry is per-compilation, so hot-reload arrives "for free" — the old registry is collected with the old compilation. The risk is the *user rewriter instance* holding state across invocations within one compilation; T02 should document rewriters as required to be stateless. T05 does not mitigate stateful rewriters.
- **Design-time vs build-time analyzers.** The IDE's `AnalyzerDriver` loads analyzers, but the `CommonCompiler.RunCore` hook only fires for command-line builds. Design-time discovery requires an IDE-side equivalent (T21 explicitly defers IDE features); for v1, `RewriterRegistry` is empty in IDE compilations and chains fall back to runtime `IQueryable` execution (T11). The diagnostic story in the IDE will lag the build by one feature iteration, intentionally.
- **Per-compilation vs cross-compilation caching.** Registry is per-`CSharpCompilation`. Assembly-level results (which AQNs resolve to which types) could be cached cross-compilation by AQN, but skipping that for v1 keeps the lifetime model simple and avoids subtle bugs where an assembly is unloaded between compilations.
- **Security implications of arbitrary analyzer-loaded code generation.** Analyzers already run arbitrary code at compile time, so loading rewriters from analyzer assemblies does not widen the attack surface for projects that have already opted into those analyzers via NuGet. However, rewriters *generate source text that is then bound at the call site* — they are strictly more powerful than diagnostic-only analyzers and warrant the `/features:compile-time-queryable` gate and an experimental warning at the public-API level. Document this in T17.
