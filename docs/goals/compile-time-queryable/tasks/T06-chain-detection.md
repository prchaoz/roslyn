# T06 — Chain detection in the binder

Identify a chain at any call whose receiver static type is the provider-marked type. Capture operator-agnostically — the compiler does not enumerate LINQ method names; any call returning the same marked queryable type belongs to the chain.

Cite — `operator-shape`: *"(a) Operator-agnostic — chain = any call returning the same marked queryable type; rewriter decides what it translates"*.

## Outcome

Two related predicates on `BoundCall`, both gated by `Compilation.IsCompileTimeQueryableEnabled` (the T04 gate):

- **Root predicate** — `IsCompileTimeQueryableChainRoot(BoundCall)`: the call's *return type* is a `NamedTypeSymbol` whose `OriginalDefinition` carries `[CompileTimeQueryable]`. Receiver may be a static factory (no receiver) or a non-chain instance call. Resolved via the T01 attribute-description lookup, mirroring `TypeSymbol.TryGetQuotableQuoterName` from the `[Quotable]` work.
- **Extend predicate** — `IsCompileTimeQueryableChainExtend(BoundCall)`: the call's *receiver* is already a `BoundCompileTimeQueryableChain` (i.e. the previous call on this receiver produced a chain object), AND the call's return type shares `OriginalDefinition` with the chain's marked queryable type. Identity is on `OriginalDefinition` so a chain transition between constructed type-arguments (e.g. `Postgres9Table<User>` → `Postgres9Table<string>` from a projecting `Select`) keeps the chain alive. Nullability annotations are ignored; mid-stream `OriginalDefinition` changes are owned by T12.

Both predicates are operator-agnostic: no LINQ method-name table, no `WellKnownMember` check on the callee. The decision is purely on receiver/return type-symbol identity. Discovery (T05) only fires when at least one predicate returns true.

When the extend predicate matches, the call is appended to the in-progress chain produced by the receiver. When the root predicate matches and the extend predicate does not, the call seeds a fresh in-progress chain with this call as its only node. The chain object is consumed by T07 (dataflow extension), T08 (terminator decision), and T09 (rewriter invocation).

## Dependencies

- **Needs T01** — the `[CompileTimeQueryable]` attribute description and well-known-type registration must exist for the predicate to read it off `NamedTypeSymbol.GetAttributes()`. Without T01 the predicate always returns false.
- **Needs T05** — discovery must be ready so that, once a chain root is detected, the rewriter assembly-qualified name can be resolved to a loaded instance (or fail with a diagnostic at the chain's binding site).
- **Gated by T04** — every entry point checks `Compilation.IsCompileTimeQueryableEnabled`. With the flag off, `TryStartOrExtendCompileTimeQueryableChain` returns false unconditionally and binding is bit-identical to today.
- **Feeds T07** — once a `BoundCompileTimeQueryableChain` exists at a method-local expression position, T07 extends it through `BoundLocal` assignments (`var q = table.Where(...); q = q.Select(...);`) via a small intra-method dataflow walker keyed on the chain's receiver-type identity.
- **Feeds T08** — terminators (`ToArrayAsync`, `foreach`, `await`, materializers, aggregates) are detected as *consumers* of a `BoundCompileTimeQueryableChain`: a `BoundCall` whose return type is *not* the marked queryable type, but whose receiver *is* the chain, ends the chain. T08 owns the exact predicate; T06 only guarantees that the chain object survives until then.
- **Feeds T09** — the rewriter is invoked at the binding site of the *terminator* (or implicit terminator, e.g. assignment to a non-chain destination, escape via return), receiving the assembled `BoundCompileTimeQueryableChain.Calls` list.

## Edge cases / risks

- **Extension methods.** Standard LINQ operators are extension methods. When `IQueryable<T> Where(this IQueryable<T> src, …)` binds against a marked source, the call's *static* return type is `IQueryable<T>`, not the marked type. The predicate must therefore allow chain continuation only when the return type is the marked type. **First-pass rule:** require strict type identity on `OriginalDefinition` (receiver and return type share the same marked `[CompileTimeQueryable]` definition). Operators that return the open `IQueryable<T>` are *not* chain-continuing — they break the chain, and the rewriter only sees the prefix. This matches the interview's "any call returning the *same* marked queryable type" reading. The provider author is expected to ship extension methods that return the marked type explicitly (e.g. `Postgres9Table<T> Where(this Postgres9Table<T>, Expression<Func<T,bool>>)`), which is also what the Postgres sample (T15) demonstrates.
- **Generic wrappers.** `Postgres9Table<TRow>` and `Postgres9Table<TProjected>` differ in type-argument but share `OriginalDefinition`. The predicate keys on `OriginalDefinition`, so `Select(row => row.Name)` returning `Postgres9Table<string>` continues a chain rooted at `Postgres9Table<User>`. The `MarkedQueryableType` field on the chain records the receiver's *constructed* type at chain-start; mid-stream changes to the type arguments are tracked per-call in `Calls`. Cross-definition transitions are owned by T12.
- **Conditional access `?.`.** `table?.Where(...)` lowers through `BoundConditionalAccess` / `BoundConditionalReceiver`. The inner access is a `BoundCall` whose receiver is a `BoundConditionalReceiver` of the marked type. The predicate must recognize `BoundConditionalReceiver` as a valid chain-receiver shape and propagate the chain object out through the surrounding `BoundConditionalAccess`. Risk: the surrounding conditional access changes the call's *effective* return type to `T?` for value types — for reference-type wrappers like `Postgres9Table<T>` the type is unchanged, so the common case works; a marked struct queryable would need a carve-out (defer to T12 if it surfaces).
- **Static vs instance receivers.** A *static* factory `Postgres9Table.From<T>(...)` returning the marked type has no receiver but is a valid chain root — the root predicate (return-type only) handles it directly. The extend predicate's "receiver is already a chain" condition is what fences off the post-root extend case from the no-receiver static case.
- **Chains starting from a method call (not a field/property).** `db.Users().Where(...)` — `db.Users()` is itself a `BoundCall` whose return type is the marked queryable type. This is the *root* case above; nothing special needed.
- **Mid-stream type changes (e.g. `IAsyncEnumerable<T>` transition).** Owned by T12. T06 surfaces the question and defers; the v1 rule (strict identity on `OriginalDefinition`) breaks the chain at the transition and emits the prefix only.
- **Cross-method dataflow.** Out of scope per `chain-boundary` ("simple local-variable dataflow within a method"). T06's chain object lives at expression positions and through local assignments only — escapes via `return`, field assignment, or method-call arguments terminate the chain (T08's call).
- **Performance.** Walking the receiver chain to attach the chain object on every `BoundCall` adds a constant per-call cost. The flag gate (T04) makes the entire path zero-cost when off. With the flag on, the predicate's `OriginalDefinition.GetAttributes()` walk reuses the existing per-symbol attribute-decoder cache on `NamedTypeSymbol` (the same site the `[Quotable]` predicate `TryGetQuotableQuoterName` uses) — symbols loaded from metadata cache their attributes once during decoding, and `SourceNamedTypeSymbol` caches via its attribute-binding state. The well-known `IsExpressionTree()` analog does *not* apply here: that check is a structural name-comparison on `WellKnownType.System_Linq_Expressions_Expression_T`, not an attribute walk, so it doesn't need caching at all.
- **Speculative binding / `SemanticModel`.** Chain objects produced inside `MemberSemanticModel`'s incremental binding must not leak into the diagnostic stream as unresolved chains. Because `BoundCompileTimeQueryableChain.DoesNotSurvive="InitialBinding"`, by the time `SemanticModel.GetDiagnostics` runs the chain is already replaced by its T09 result (rewriter output or fallback). Mid-binding cancellation is the residual risk; assert that any chain not replaced by end-of-binding is unwrapped to its last `BoundCall` so consumers never observe the synthetic node.

## Open questions

- **T12 (IAsyncEnumerable scope).** Does the "receiver static type is marked" rule generalize when a chain transitions mid-stream into `IAsyncEnumerable<T>`? E.g. `pgTable.Where(...).AsAsyncEnumerable().WhereAwait(...)` — does the second `Where` continue the chain, or does the transition split it? T06's v1 answer is "strict identity, chain breaks at the transition." T12 may relax this; if so, the predicate gains a "type-equivalence" mode (provider-declared via attribute parameter). Surfaced only.
- **T13 (branched chains).** A local consumed by multiple downstream calls — `var q = table.Where(...); var a = q.ToArray(); var b = q.Select(...).ToArray();` — produces a DAG, not a list. T06 currently appends to a single `Calls` array per chain object; reuse of the local in two terminators means each terminator sees the *same* prefix chain via T07's dataflow propagation. Whether the rewriter sees one chain per terminator (linear, with shared prefix duplicated) or one DAG with shared structure is the T13 decision. The `BoundCompileTimeQueryableChain` field set may need to grow accordingly. Surfaced only.
